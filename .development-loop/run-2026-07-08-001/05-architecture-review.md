# 05 — Architecture Review

Reviewed: `04-spec.md`, cross-checked against `03-architecture-decision-corrected.md`, `03-risk-assessment.md`, `01-intake.md`, and the actual current code (`web/src/hooks/useChat.ts`, `web/src/lib/agentStudio.ts`, `scripts/agents/build_acs_agents.mjs`, `web/src/types.ts`, `web/src/components/DiscoveryCard.tsx`). No code changes exist yet — this is a pre-implementation gate.

## Findings

### Critical

**1. `deepDiveQuery` contract is stated in prose, not pinned — and the call chain it feeds has a real duplication risk the spec never walks through.**
Spec (line ~57): *"replaced by `ChatTurn.deepDiveQuery?: string` (the resolved query to send Technical, now just the user's original turn text + Generic's answer...)"*. This sentence is ambiguous about whether `deepDiveQuery` is (a) just `turn.query` (the fallback path that already exists today at `useChat.ts:316`, `turn.pendingToolCall?.query || turn.query`), or (b) a literal concatenation of the question and Generic's answer into one string. That distinction matters mechanically: `runTechnicalLeg`'s `query` parameter is used twice — once folded into `technicalHistory` as `{role:'user', content: query}` (`useChat.ts:236`), and again passed as `req.query` to `callCompletions`, which appends *another* `{role:'user', content: req.query}` (`agentStudio.ts:183`). If `deepDiveQuery` ever became the question+answer concatenation, Technical's final "user" turn would literally contain Generic's answer text framed as something the user said. The spec's other three contracts (PATCH body, frame type, `extractDeepDiveOffer`) all get a real before/after code block; this one — the field that actually decides what gets sent to the specialist agent — does not. It needs one, and it needs to state explicitly that `deepDiveQuery = turn.query` (matching today's existing fallback), with the answer text staying where it already lives (the separate `genericText` parameter), not folded into the same field.

**2. The single most load-bearing code location in `useChat.ts` is never rewritten in the spec.**
Today, one state update (`useChat.ts:206–210`) sets `deepDiveOffered`, `followUp`, and `pendingToolCall` together from the tool-call detection. The spec's "detection contract swap" section shows `extractDeepDiveOffer` being added and `pendingToolCallFrom`/`FOLLOWUP_RE` being removed, but never shows the replacement for that specific `setTurns` call — i.e., how `deepDiveOffered`, `followUp` (from `rest`), and the new `deepDiveQuery` actually get assigned from `extractDeepDiveOffer`'s return value. This is exactly the seam the risk assessment worried about ("silent feature failure... ships looking done with no error"): if Build improvises this wiring without a spec'd contract, it's the highest-probability place for a quiet mismatch (e.g., `deepDiveOffered` staying true while `deepDiveQuery` is undefined, or vice versa).

**3. Go-condition #4 (dry-run against dev/disposable agents) has no mechanism in the spec, and none exists in the script today.**
`build_acs_agents.mjs`'s `PERSONAS` array hardcodes agent *names* (`ACS-generic-neural`, `ACS-technical-neural`, lines 49–50), and `listAgents()` resolves those names straight to their live IDs before every PATCH — there is no flag, env var, or name-suffix mechanism to target a disposable/dev copy instead. Risk finding #4 and Go-condition #4 both require a dry-run against dev/disposable agents *before* touching the live-linked IDs in `spectrum.ts`, but the spec's PATCH-body section only describes the target contract (what `suggestions` config to push) — it never adds the dry-run mechanism itself (e.g., an `ACS_AGENT_SUFFIX` env var, a `--target=dev` flag, or a documented manual step of cloning the two agents under temporary names first). As written, the only way to "dry-run" is to hand-edit `PERSONAS` before every run and hand-edit it back — which is exactly the kind of manual discipline that produced this session's earlier live-agent incident. This Go-condition is not concretely specified; it's gestured at by cross-reference to "the corrected architecture's own precedent" without translating into an actual script contract.

### Important

**4. Go-condition #3 (test coverage as a Build gate) has no concrete plan — it's the thinnest-covered of the four.**
The spec has no dedicated test-plan section. Compare the treatment of the other three Go-conditions: #1 gets an exact PATCH-body diff plus a verification assertion; #2 gets "must empirically capture a real completions response... before writing the parser branch" plus an explicit instruction not to let it fall through the catch-all. #3 gets nothing beyond the Go-condition citation itself — no named test files, no listed cases (e.g., "test: `extractDeepDiveOffer` isolates a `SPECIALIST:` prefix and returns the rest," "test: `parseCompletionStream` on a captured suggestions frame yields `ParsedCompletion.suggestions` non-empty"), no coverage target. Given `useChat.ts`/`agentStudio.ts` have zero existing tests (confirmed by the risk assessment's own repo search) and are exactly the files being rewired, this is the item most likely to silently slip from "Build gate" to "we'll get to it," which is the risk assessment's own named failure mode.

**5. Rollback strategy doesn't address PATCH atomicity, which the risk assessment explicitly flagged as unverified.**
`04-spec.md`'s Rollback section is correct about the mechanism (PATCH-in-place means no ID churn, `git revert` covers the frontend side, nothing stateful to unwind) — genuinely sound for what it covers. But `03-risk-assessment.md` (component 2, Denial of Service row) separately flagged that "whether Algolia's PATCH endpoint does a partial merge or full-object replace is unverified... does a 422 leave the agent config unchanged, or partially applied?" The spec's rollback plan never revisits this. If a PATCH to a live agent partially applies before a validation error (e.g., `config.suggestions` lands but `tools` doesn't, because `extraTools` was malformed), "just revert the script and re-run" assumes the *next* PATCH will cleanly overwrite the partial state — plausible, but unverified, and not called out as a residual assumption in the Rollback section where a reader would expect it.

**6. The PATCH-body diff's `tools` line comment is slightly misleading about where R12 actually lives.**
The diff shows `tools, // Generic's tools no longer includes CONSULT_TECHNICAL_TOOL (R12)` as if the change is visible at the PATCH-body construction site. In the real script, `tools` is computed upstream as `[...scopeTools(base.tools, filters, desc), ...extraTools]` (`build_acs_agents.mjs:82`), and the actual edit R12 requires is `PERSONAS[0].extraTools: [CONSULT_TECHNICAL_TOOL]` → `extraTools: []` (line 49). The spec's prose ("R12: remove `consult_technical_specialist` from Generic's `extraTools`") does correctly name the real edit site — but the code diff right above it points at the wrong line, which is exactly the kind of small mismatch that causes a Builder to edit the PATCH body instead of `PERSONAS`. Fix the diff to point at line 49, not line 92.

### Minor / things the spec gets right (no change needed)

**7. Scope discipline on scalability is correct, not a gap.** The spec doesn't re-litigate the risk assessment's unbounded-history/added-LLM-call concern (pre-mortem row 6, STRIDE component 4 DoS row) — it's out of scope per Intake #4 (the `<Chat>` widget migration is where rate-limiting belongs) and the risk assessment itself said "accept for this scope's actual traffic profile." Silently deferring would be a problem; the spec inherits an already-explicit acceptance, which is the right move.

**8. Security posture is consistent with the STRIDE findings — no drift.** The spec introduces no new trust boundary and doesn't attempt to "fix" the fabricated-history / public-search-key exposure the risk assessment named and explicitly accepted (component 3/4 Spoofing rows). Good — a spec that quietly added partial mitigation here without being asked would be scope creep; this one doesn't.

**9. Coupling across the three touched files is clean.** `build_acs_agents.mjs` (config push) → `agentStudio.ts` (pure wire parsing) → `useChat.ts` (orchestration/UI state) is a one-directional pipeline that matches the existing layering; the spec doesn't introduce any new cross-file dependency or shared mutable state beyond what's already there. No cohesion complaint.

**10. R12 itself is a genuinely good catch, grounded in the actual code, not the plan docs.** The spec correctly identifies that `useChat.ts`'s own comment ("Generic sometimes calls the tool with no acknowledgment first") describes a *live* production defect, not a hypothetical, and that removing the prose-sentinel parsing alone wouldn't fix it — the tool registration itself has to go. This is the strongest piece of the spec.

**11. `context.include_tool_outputs: true` and the grounding-continuity language ported into `system_prompt` are correctly deferred, not invented.** The spec doesn't try to write the actual prompt wording (correctly punts to the vault's Open Question #6) and explicitly preserves the existing grounding rule from `instructions_technical.md` rather than re-deriving it. Right call.

## Resolution (2026-07-09)
All 3 Critical + 3 Important findings applied directly to `04-spec.md`:
1. `deepDiveQuery` pinned to `turn.query` alone, with the double-user-turn risk named explicitly.
2. Exact `setTurns` rewiring at the `useChat.ts:206–210` seam added, deriving `deepDiveOffered`/`deepDiveQuery` from the same `offer` value so they can't disagree.
3. Dry-run mechanism added (`ACS_AGENT_SUFFIX` env var) — real script contract, not a hand-edit-and-revert discipline.
4. Named test-case list added (5 cases covering `extractDeepDiveOffer`, frame parsing, state wiring, the double-user-turn regression, and R12 regression).
5. Rollback section now names the PATCH-atomicity assumption explicitly and explains why the rollback plan doesn't depend on resolving it.
6. PATCH-body diff comment corrected to point at `PERSONAS[0].extraTools` (line 49), not the PATCH construction site.

## Verdict

**Approve with changes — not a rework, but not ready to hand to Build as-is.**

The architectural shape is sound: collapsing Job A/Job B into one native `suggestions` mechanism is a real simplification over the tool-call design, the three touched files are the right seam, and the spec correctly stayed out of scope creep on scalability/security. Two of the risk assessment's four Go-conditional items, however, are gestured at rather than concretely specified — test plan (#3) has no named test cases at all, and dry-run-before-live (#4) has no mechanism in a script that today can *only* target the live agent names. Those need to be turned into real contracts (specific test cases; an actual dry-run flag/env var in `build_acs_agents.mjs`) before Build starts, not discovered mid-build. Separately, the `deepDiveQuery` field — the one piece of new state that decides what Technical actually receives — needs its value and the exact `useChat.ts:206–210` rewiring pinned down with a code diff, the same rigor already given to the other three contracts in this spec. None of this requires re-deriving the architecture; it's tightening three sections of the same document.
