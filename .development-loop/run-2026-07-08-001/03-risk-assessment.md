# 03 — Risk Assessment

Scope: the corrected design in `03-architecture-decision-corrected.md` — native `config.suggestions` on both agents (replacing the tool-call handoff + `[[FOLLOWUP:...]]` sentinel), the frontend's direct Technical-call context-pass (unchanged mechanism, new trigger), and new multi-turn history accumulation (R11). Findings below are grounded in the code as it stands today (pre-change baseline) — `web/src/hooks/useChat.ts`, `web/src/lib/agentStudio.ts`, `web/src/lib/agents.ts`, `web/src/components/DiscoveryCard.tsx`, `web/src/types.ts`, `scripts/agents/build_acs_agents.mjs`, `web/src/config/instances/spectrum.ts`, `vercel.json` — not re-derived from the plan docs alone.

## STRIDE per component

### 1. Client-side suggestion-signal parsing
Replaces `FOLLOWUP_RE` regex scanning (`useChat.ts` lines 35-44) and the tool-call detection path (`pendingToolCallFrom`, lines 46-55) with reading a native suggestions frame/part; `DiscoveryCard.tsx` gets rewired from `followUp` (regex-sourced) to the native field (R10).

- **Spoofing:** N/A as a distinct concern — the suggestions payload originates from Algolia's own Agent Studio backend over HTTPS to `{appId}.algolia.net`, the same trust boundary as every other frame this client already parses.
- **Tampering:** `parseCompletionStream` (`agentStudio.ts`) has a "skip malformed frame, never throw" discipline for known prefixes (`0`/`9`/`a`/`3`) — but any new suggestions frame type is currently NOT in that explicit list, so it falls into the generic ignore-and-continue branch (lines 139-141). That's safe from a crash standpoint but means a malformed *or entirely valid* suggestions frame is silently dropped either way until code is added to actually read it. This is a correctness gap more than a security one, but it means "the mitigation" (native parsing replacing the regex hack) does not exist in the code yet.
- **Repudiation:** N/A — no accountability requirement for an internal demo tool.
- **Information Disclosure:** `context.include_tool_outputs: true` (R9) feeds that turn's retrieved search-hit content into the suggestion-generation call. Since the corpus is 100% public Adobe Spectrum documentation (confirmed: `ACS_SPECTRUM_MULTI`, self-fetched public docs, no PII per Intake #5/#6), there is no sensitive-data disclosure risk in what gets surfaced back to the user as a suggestion.
- **Denial of Service:** enabling `suggestions` on both agents adds one additional `gemini-2.5-flash-lite` call after every completions call, on every turn, with no client-side cap — see pre-mortem row 6.
- **Elevation of Privilege:** N/A — this component only renders/consumes data; it introduces no new access boundary.

### 2. Agent config push mechanism (`scripts/agents/build_acs_agents.mjs`)
The mechanism that will PATCH `config.suggestions` onto `ACS-generic-neural`/`ACS-technical-neural`. This exact file is where this session's real incident happened (delete+recreate silently breaking a live, ID-hardcoded agent) — already fixed to PATCH-in-place (lines 84-95, confirmed live 2026-07-09 per the code's own comment).

- **Spoofing:** authenticates via `ALGOLIA_ADMIN_API_KEY` read from `.env.local` (lines 16-18) — unchanged mechanism, admin key never reaches the browser (confirmed: `agents.ts`/`agentStudio.ts` only ever construct configs from `VITE_ALGOLIA_SEARCH_API_KEY`).
- **Tampering:** the PATCH body is currently hardcoded to `{instructions, model, providerId, tools}` (line 92) — it does **not** yet include a `suggestions` field. Until this is added, the mitigation this change depends on (pushing suggestions config live) simply isn't wired — see pre-mortem row 1. Separately: whether Algolia's PATCH endpoint does a partial merge or full-object replace is unverified; if it's a full replace, an incomplete future edit that adds `suggestions` to one persona's construction but not the other reproduces the *same root-cause shape* as the incident this session already fixed once (a field silently absent from what gets pushed).
- **Repudiation:** the script prints a per-persona result line (old/new ID, patched-in-place note, index/filter/tools summary — lines 104-105) to the terminal only; there's no persisted diff or log file. Sufficient for a human operator watching the run, insufficient as an automated detector if a future run silently regresses a field — nothing re-checks "did `suggestions` come back non-null" the way `index`/`filter`/`tools` are already echoed.
- **Information Disclosure:** N/A — build-time Node script, requires local possession of the admin key; never runs in a browser context.
- **Denial of Service:** PATCHing a live, published, actively-demoed agent mid-session; the script already guards non-200 responses with `process.exit(1)` (line 94), but atomicity of a rejected PATCH (does a 422 leave the agent config unchanged, or partially applied?) is unverified. The corrected architecture's own precedent (dry-run against disposable dev/spike agent copies before touching the live-linked IDs in `spectrum.ts`) is the stated mitigation — it must actually be followed for this change, since it changes a different config surface (`suggestions`) than the tool-call spike that established that discipline.
- **Elevation of Privilege:** N/A — same admin-key-scoped operation as before; adding a config field to an already-privileged PATCH introduces no new privilege boundary.

### 3. Direct Technical-call context-passing
Already exists in code today (`runTechnicalLeg`, `useChat.ts` lines 221-302): frontend calls Technical's `/completions` directly, folding the original question + Generic's answer text into `technicalHistory` (lines 234-238) as a synthesized `{role:'assistant', ...}` turn. The corrected architecture keeps this mechanism unchanged and only changes what *triggers* it (a native suggestion offer instead of a tool-call pause).

- **Spoofing:** the frontend asserts to Technical "the assistant already said X" by constructing that history entry itself — Technical has no independent way to verify Generic actually produced that text. This is a pre-existing property of the mechanism (not introduced by this change), but worth naming explicitly rather than silently inheriting: anyone who can reach the completions endpoint directly (the search-only key is public by design) can fabricate arbitrary prior "assistant" turns.
- **Tampering:** history arrays are plain mutable client-side JS with no integrity check (no signature, no server-side reconciliation) binding a "past turn" to something an agent actually emitted. Accepted for this tool's threat model (no adversarial-user assumption per Intake #2/#5) but should be a named, not silent, acceptance.
- **Repudiation:** N/A.
- **Information Disclosure:** Generic's full answer text is forwarded to Technical as context on every deep-dive — since both agents read the same public corpus, this introduces no new disclosure.
- **Denial of Service:** no size cap on `genericText` before it's folded into `technicalHistory`; a long Generic answer becomes part of every subsequent Technical call's payload for the rest of the conversation (compounds with component 4, since `/completions` is stateless and resends full history every call per `docs/workspace/ac-chat-ux/00-protocol-read-receipt.md`).
- **Elevation of Privilege:** N/A.

### 4. Multi-turn history accumulation (R11 — not yet built)
Decided design (Architecture doc lines 28-30): `genericHistory`/`technicalHistory` accumulate across rounds; on the 2nd+ deep dive, Technical gets all past questions **verbatim** plus a **summary** (mechanism undecided — client truncation vs. agent-emitted) of each past answer.

- **Spoofing:** compounds component 3's fabricated-history exposure across N rounds — one bad or injected turn early in a long conversation gets re-sent and implicitly trusted on every later round, since nothing re-validates accumulated history against ground truth.
- **Tampering:** the summarization mechanism is explicitly undecided as of this document. If it ends up being agent-emitted (the agent produces its own short summary of its answer), that summary is untrusted-if-hallucinated content that then compounds into future turns' premises — a grounding-integrity risk specific to this project's "110% grounded, every claim traceable" standing rule, not just a generic security concern: a drifted summary lets a later turn build on an ungrounded premise while the UI still shows grounded-looking source cards for the *current* turn only.
- **Repudiation:** N/A.
- **Information Disclosure:** no new disclosure surface from the public corpus itself; however, `AnswerSegment.rawHits` (`types.ts` lines 42-48) is explicitly documented as judge-only, larger/rawer than what should travel to another agent as "prior context" — whatever code builds the new history-accumulation helpers at Build time must keep using `segment.text` (already-parsed display text), never `rawHits`, or an unrelated field meant for the offline judge starts round-tripping into live agent-to-agent requests.
- **Denial of Service:** the core new risk. R11 states past questions travel verbatim (unbounded) with no stated cap on number of rounds or total history size. Combined with component 1's added per-turn suggestion call and the public search-only key (any visitor to the deployed Vercel URL can script requests directly against the completions endpoint, bypassing the UI's own round-count), this is a real but currently low-traffic-bounded cost/availability exposure (see pre-mortem row 6).
- **Elevation of Privilege:** N/A.

## Compliance gap analysis

**Conclusion: N/A — stated explicitly, not skipped.**

Per Intake #5: "none apply — internal sales-demo tool, no PII/PCI/HIPAA/GDPR-regulated data in the corpus (Adobe Spectrum public docs)." This assessment independently checked that claim against the code rather than accepting it at face value:

- The corpus (`ACS_SPECTRUM_MULTI`) is self-fetched public Adobe Spectrum documentation — confirmed via the project's own `CLAUDE.md` ("Corpus is LIVE... 502 recs... self-fetch") and this doc's file reads; no ingestion path touches user data.
- The data model carried through the app (`types.ts`: `ChatTurn`, `AnswerSegment`, `HistoryEntry`, `AnswerSource`) holds only question/answer text and Algolia-doc titles/URLs/source facets — no user identifiers, accounts, emails, session cookies, or forms were found anywhere in the read files.
- There is no authentication layer, no user accounts, and no server-side persistence of conversations — every conversation lives in browser React state only (`useState`/`useRef` in `useChat.ts`) and disappears on `reset()`/page reload.
- The only credential in the browser is a deliberately search-only Algolia key (`agents.ts` explicitly throws if the admin key path is used instead) — a security-hygiene property, not a regulated-data compliance matter.

So GDPR/PCI/HIPAA are correctly N/A for this change's actual scope. Forward-looking note (non-blocking): if this tool later gains a real login/account layer, or analytics/telemetry that ties free-text queries to an identifiable visitor, that would reopen a privacy question this assessment does not need to answer today — flag for whoever scopes the `<Chat>` widget migration (explicitly out of scope per Intake #4).

## Go/No-Go

**Go, conditional on four Build-stage items being treated as acceptance criteria, not assumed:**

1. `build_acs_agents.mjs`'s PATCH body must actually include `suggestions`, with the script's existing post-PATCH verification print extended to echo it back non-null (mirrors the exact incident-class already fixed once this session for other fields).
2. `agentStudio.ts` must explicitly handle the new suggestions frame type (not rely on the "ignore unknown frames" catch-all), with a test proving the round-trip into `DiscoveryCard`.
3. NR3's test requirement must be honored literally: `useChat.ts`/`agentStudio.ts` currently have zero test coverage of any kind, and they are the exact modules this change rewires — tests land WITH Build, per this project's own TDD mandate, not deferred.
4. The PATCH must be dry-run against disposable/dev agent copies before touching the live-linked IDs in `spectrum.ts` — the same discipline the corrected architecture's own history already established, applied to this new config surface.

None of the findings above are irreversible, involve regulated data, or constitute a security breach in the classic sense — this is a low-blast-radius internal/prospect-demo tool with no PII, no auth boundary, and a corpus that's already public. The two live findings that matter most are functional/operational, not confidentiality: (a) repeating the incident *class* from earlier this session (a config field silently missing from a push) if `suggestions` isn't added to the PATCH body and verified, and (b) shipping a silently-broken feature if the new SSE frame type isn't explicitly parsed. Both are ordinary Build-stage engineering gates, not architectural blockers — proceed to Spec/Build with these four items named up front.
