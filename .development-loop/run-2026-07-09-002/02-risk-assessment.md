# 02 — Risk Assessment

Scope: both tracks of `docs/plans/2026-07-10-reconciled-handoff-architecture-build.md` — Track A (replace Agent Studio's racy async `config.suggestions` job with a client-side, synchronously-awaited classification call as the `SPECIALIST:` offer signal) and Track B (a third live orchestrator agent with 2 `client_side` tools, real tool-call frame interception/resume, auto-chained specialist calls that remove the human "yes, go deeper" consent gate). Findings below are grounded in the code as it stands **today** (pre-change baseline) — `web/src/hooks/useChat.ts`, `web/src/lib/agentStudio.ts`, `web/src/lib/agents.ts`, `web/src/config/instance.ts`, `web/src/config/active.ts`, `web/src/config/instances/spectrum.ts`, `web/src/types.ts`, `scripts/agents/agentConfig.mjs`, `scripts/agents/build_acs_agents.mjs`, and the three already-executed spike scripts in `scripts/spikes/agent-tool-handoff/` (`create-probe-agent.mjs`, `probe-orchestrator.mjs`, `e2e-orchestrator-validation.mjs`) plus their captured wire evidence (`docs/spikes/orchestrator-round2-frames.txt`, `docs/spikes/2026-07-10-e2e-orchestrator-results.json`) and `docs/spikes/2026-07-09-pure-orchestrator-VERDICT.md` — not re-derived from the plan doc's prose alone. Neither track has any implementation code written yet (confirmed: `agentConfig.mjs`'s `PERSONAS` array still lists exactly 2 personas, `useChat.ts` still reads `genericResult.suggestions` directly, `spectrum.ts`'s `agents` object still has exactly `generic`/`technical`) — this assessment is pre-Build, as expected for the RISK stage.

## STRIDE per component

### 1. Track A — client-side classification call (`classifyOffer`, not yet built)
Replaces `runTurn`'s dependence on `genericResult.suggestions` (`useChat.ts:283`, currently sourced from the racy async `config.suggestions` platform job) with a synchronous, client-awaited `completions` call using `suggestions_generic.md`'s content, per `build.md:33-48`.

- **Spoofing:** N/A — same trust boundary as every other call this client already makes: HTTPS to `{appId}.algolia.net`, search-only key (`agents.ts:42-45`).
- **Tampering:** Bounded by construction, not just intent: `deriveOfferState`'s contract (which Track A must keep "byte-for-byte unchanged," `build.md:53`) only ever lets the classification call's output toggle `deepDiveOffered`/`followUp` — it never supplies the text sent onward to Technical (`deepDiveQuery` stays `turnQuery` verbatim per `deriveOfferState`, `useChat.ts:54-64`, and per `types.ts:77-81`'s own documented invariant). Even a fully adversarial classification response can only suppress/force the offer or poison the `followUp` discovery-card text (low severity, self-inflicted by the querying user) — it structurally cannot redirect what question actually reaches Technical. This is the sharp contrast with Track B's finding below, where the analogous value (the tool call's `args.query`) is proposed to feed the ACTUAL call, not just a boolean.
- **Repudiation:** N/A.
- **Information Disclosure — the two specific questions asked of this assessment:**
  - *Could a crafted query make the classification call leak the admin key?* No — structurally impossible, not just unlikely. `getEnvConfig()` (`agents.ts:25-38`) is the ONLY source of credentials any browser code can construct a `CompletionsConfig` from, and it reads exactly `VITE_ALGOLIA_APP_ID`/`VITE_ALGOLIA_SEARCH_API_KEY` — confirmed by grep across `web/src/**` that no `ADMIN`-named variable is referenced anywhere in client code, and no `VITE_*ADMIN*` variable exists in any source/config file in the repo. `ALGOLIA_ADMIN_API_KEY` is read only by Node scripts (`build_acs_agents.mjs`, crawler/eval scripts, spike scripts) via direct `.env.local` file parsing — never through `import.meta.env`, never bundled. The admin key is not "in the browser but protected" — it is never present in browser memory at all, so there is nothing for a prompt-injected response to exfiltrate through this path.
  - *Could a crafted query leak the classification prompt (`suggestions_generic.md`) or Generic's system prompt?* Only to the same degree any user can already attempt to extract Generic's/Technical's own system instructions today via ordinary prompt-injection — not a new threat class. Since A1-A4 add no new agent, the classification content is folded into a message sent to an *existing* agent whose own hardened grounding instructions (the project's standing "110% grounded" rule) are the thing constraining leakage, exactly as they already do for the main answering role. No new trust boundary is introduced.
- **Denial of Service:** Adds one additional awaited `completions` call per turn (not per deep-dive — per ordinary turn), increasing latency before the offer/discovery-card renders (Generic's streamed answer text is unaffected — `onText` still streams independently). No new client-side rate limit exists; same accepted posture as the prior assessment's equivalent finding.
- **Elevation of Privilege:** N/A.

- **Spoofing/Tampering/Repudiation/Info Disclosure/Elevation:** N/A — no code path changes here for the required tasks.

### 3. Track B — orchestrator agent creation (`ACS_AGENT_SUFFIX` dry-run discipline)
The specific question this assessment was asked to answer: is there any code path in either track that could write to an agent name without the suffix guard?

- **Spoofing:** N/A.
- **Tampering — the guard itself:** `buildAgentName(name, SUFFIX)` (`agentConfig.mjs:36-38`) is applied at every lookup/create/PATCH site in `build_acs_agents.mjs` (`names` at line 44, `agentName` at line 64) — confirmed this is the ONLY code path that writes any `ACS-*`-named agent in the current repo. Grepping every script under `scripts/` for hardcoded `'ACS-`/`"ACS-` names that also perform a POST/PATCH/DELETE returns nothing outside `agentConfig.mjs`/`build_acs_agents.mjs` themselves. **Conclusion: CLOSED for the documented Build path** — Track B's task B1 ("Add orchestrator persona + tools to `agentConfig.mjs`") explicitly routes the third agent through this same guarded mechanism, provided it's actually implemented that way (see Go-condition below).
  - **The prospective risk (not yet realized):** every already-executed spike script (`create-probe-agent.mjs:76`, `probe-orchestrator.mjs:34,103`, `e2e-orchestrator-validation.mjs:42,131`) creates/deletes agents via raw `fetch` + the admin key, using its own hardcoded disposable name (`SPIKE-tool-probe`, `SPIKE-orchestrator-probe`, `SPIKE-orchestrator-e2e`) — completely outside `buildAgentName`. This is safe today (none of them ever reference an `ACS-*` name in a write call — confirmed by grep, and `VERDICT.md:18,38` documents this as a deliberate constraint, checked "by construction"). It stops being safe the moment someone adapts one of these scripts' create/resume logic directly into a script or one-off that writes an `ACS-`-namespaced agent without going through `buildAgentName`+`SUFFIX`. The plan doc itself flags the risk of copy-pasting these scripts (`build.md:87`) but frames it as a code-quality note, not a security control — this assessment reclassifies it as one.
  - **A separate, smaller drift note found while checking this:** `create-probe-agent.mjs:71` hardcodes `baseId = '13809d4b-6b6d-4297-b95c-a934bceef0b4' // ACS-generic-neural`, which is the PRE-incident ID (Generic's live ID today, per `spectrum.ts:31` and `CLAUDE.md`, is `95826da6-...` — the old ID was deleted server-side by the since-fixed script bug referenced in `CLAUDE.md`'s agent-namespace section). This script only does a read-only `GET` on that stale ID (harmless — it 404s or reads nothing), but it is exactly the kind of stale-hardcoded-ID drift this project has already been burned by once; worth a cleanup pass, not a blocker.
- **Repudiation:** `build_acs_agents.mjs`'s per-persona print (`build_acs_agents.mjs:91-93`) already extends to a third persona automatically once it's added to `PERSONAS` — no gap here.
- **Information Disclosure:** N/A — admin-key-gated, Node-only, same as today.
- **Denial of Service:** N/A beyond what's already covered in the pre-mortem (scopeTools gap).
- **Elevation of Privilege:** N/A for the agent-creation mechanism itself (same admin-key-scoped operation).

### 4. Track B — tool-call frame interception + resume mechanism (`agentStudio.ts` / `useChat.ts`, not yet built)
The specific question this assessment was asked to answer: does auto-chaining on a model-decided tool call create a new injection surface versus today's direct-call pattern?

- **Spoofing:** The orchestrator's tool-call `args` are LLM output, not a value the client independently verifies against the human's actual input. Today's direct-call pattern has zero LLM hops between "what the human typed" and "what gets sent to Generic/Technical" — `runTurn`/`runTechnicalLeg` always use `query`/`turn.query`/`turn.deepDiveQuery`, all of which trace verbatim back to `sendMessage`'s `trimmed` (`useChat.ts:391-405`). Track B's plan, as literally written (`build.md:78`, "call REAL Generic directly with the tool call's `query` arg"), inserts exactly one such hop for the FIRST leg and (by the same described pattern) implicitly for the specialist leg too. **This is the answer to the question asked: yes, this is a materially new mechanism versus today's direct-call pattern** — see the pre-mortem's top row for the full analysis and required mitigation. Rated HIGH given Track B's auto-chain removes the human review step that would otherwise catch a reformulated/injected query before Technical ever sees it.
- **Tampering:** Same finding, restated at the history layer — if a rewritten `runTurn` feeds the tool-call's `args.query` (rather than `turn.query`) into `buildTechnicalHistory` (`useChat.ts:71-81`) or into what later becomes `turnToHistory` (`useChat.ts:142-149`), a single successful reformulation contaminates every subsequent turn's replayed history, compounding exactly the way the prior assessment's "fabricated history" finding already described — except now the fabrication vector is a live LLM decision reachable from ordinary chat input, not just someone scripting raw HTTP against the public key.
- **Repudiation:** N/A.
- **Information Disclosure:** No new disclosure path beyond what §1 already covers (public corpus, no admin key reachable). The orchestrator's own instructions ("never answer, only route") are a new system-prompt surface, but subject to the same accepted extraction-attempt exposure every agent already has.
- **Denial of Service:** Auto-chaining converts the deep-dive leg from opt-in (human click) to unconditional-on-model-decision — see pre-mortem row 6.
- **Elevation of Privilege:** Framed precisely: the orchestrator LLM is granted the effective authority to trigger a live Technical call that previously required an explicit human action. Intake and `01-intake.md` are clear this UX change itself is already approved and out of scope to re-litigate — this assessment does not re-open "auto-chain vs. gate." What IS in scope, and unmitigated in the plan as written, is the query-provenance question above: *given* auto-chain is approved, the text that reaches Technical on that automatic path must still be the value the human actually typed, not a value the newly-elevated orchestrator hop is free to alter.

- **Repudiation/Info Disclosure/Elevation:** N/A.

## Go/No-Go

### Track A — Go, conditional on three items being treated as acceptance criteria:
1. `runTurn`'s classification call must fully REPLACE `genericResult.suggestions` as `deriveOfferState`'s input, never merge with it (pre-mortem row 5) — verified by A1's RED test asserting the exact input `deriveOfferState` receives, not just its output.
2. The classification call's target agent and exact request shape must be named explicitly in the implementation and covered by a test (pre-mortem row 4) — currently an inference (Generic's own existing `agentId`), not a written decision.
3. A4's repeated-query live probe (already in the plan) is the real acceptance bar — a green test suite alone does not close this track, per this project's own Cardinal Rule and the plan's own bug-fix loop.

None of Track A's findings are Critical or High. The classification call is well-bounded by the existing `deriveOfferState` contract (§1 above) — it can affect whether/what discovery-card text renders, never what question reaches Technical.

### Track B — Go, conditional on five items being treated as acceptance criteria (two of these are new HIGH findings this assessment surfaced, not restatements of the plan's own task list):

1. **(HIGH, new)** `runTurn`'s rewrite must call Generic/Technical with the client's own trusted `turn.query`/`deepDiveQuery` value — never the orchestrator tool call's `args.query` — with a test that feeds a tool-call frame whose `args.query` differs from the turn's real query and asserts the real call still uses the turn's query. This directly contradicts the plan doc's literal current wording (`build.md:78`) and must be corrected before or during B3, not discovered in review.
2. **(HIGH, new)** `build_acs_agents.mjs`'s `scopeTools` must gain an explicit "no search tool" path before the orchestrator persona is added via B1 — without it, the orchestrator silently inherits search capability that breaks its "never answers, never searches, only routes" design premise the spike actually validated.
3. **(MEDIUM, new)** B4/B5's "-dev deploy" live verification needs real plumbing (an env-driven agent-ID override in the frontend, or an explicitly reviewed branch-diff procedure) before it can be run safely — none exists today, and the path of least resistance (hand-editing `spectrum.ts`) repeats this project's own prior incident shape.
4. **(Restated from the plan's own intent, now made checkable)** The orchestrator MUST be added via `agentConfig.mjs`'s `PERSONAS` array / `buildAgentName`+`ACS_AGENT_SUFFIX`, not via adaptation of the spike scripts' raw admin-key `fetch` pattern — `build.md:87`'s "not copy-pasted" line is a security control, not just a code-quality note, and should be verified in code review as such.
5. B6's human-gated production flip (already in the plan) stays as the final go/no-go for touching the live-hardcoded IDs in `spectrum.ts` — unchanged requirement, restated for completeness.

**Nothing found here is a data-loss, compliance, or credential-exposure risk** — the admin key structurally never reaches the browser regardless of either track (§1), and the corpus remains public with no PII path (compliance section above). The two HIGH findings are about **trust provenance of text reaching a live, auto-triggered agent** and **an architecture-breaking gap in the only existing build-script mechanism for the orchestrator's tool scope** — both are concrete, code-level, fixable before Build, and neither is a reason to abandon Track B; they are reasons not to implement it exactly as the plan doc's prose currently describes it. Proceed to Build for Track A now; proceed to Build for Track B only after Track A ships and is verified live (per Intake's sequencing) and with the five items above named as explicit tasks, not assumptions.
