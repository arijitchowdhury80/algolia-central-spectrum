# Build document: reconciled Generic→Technical handoff architecture

Status: PLAN ONLY — nothing in this document has been executed. Supersedes `docs/plans/2026-07-09-revert-to-tool-call-and-port-to-ac2.md` Phase 0 (Phase 1, the AC2 port, is untouched by this doc and still applies once Phase 0's replacement is decided).

Written after a real E2E test (`scripts/spikes/agent-tool-handoff/e2e-orchestrator-validation.mjs`, results in `docs/spikes/2026-07-10-e2e-orchestrator-results.json`) validated the pure-orchestrator design on real data: 10/10 real offer-signal precision, 10/10 real orchestrator-chaining-correctness, 5/5 real specialist answers. The test passed. This document exists because a passing test is not the same question as "should we build this."

## 0. Read this first — the finding that changes the plan

The 2026-07-09 revert plan's whole reason for existing was one caveat, logged the same night in `SESSION.md`:

> "the suggestion is generated by a *separate async* `gemini-2.5-flash-lite` completion that races Agent Studio's per-query cache. Some implementation queries land in a suggestion-less cached state and then **never** show the offer on the single completions stream the client reads."

That's the actual production bug: the `SPECIALIST:` offer sometimes silently doesn't appear on implementation questions, because Agent Studio generates it as a second, platform-internal async job that can lose a race against its own response cache.

Tonight's E2E test validated that a **3-agent orchestrator** can correctly chain a tool call based on an `offerSpecialist` signal. But the signal I fed it — in every one of the 10 real trials — was `ACS-generic-neural-dev`'s own real `config.suggestions` output. **Same async mechanism, same cache, same race.** The orchestrator sits entirely downstream of the exact thing that's unreliable. Building it does not touch the caching race. It adds a third agent, real tool-call plumbing (`agentStudio.ts` frame parsing, `useChat.ts` pause/resume state, agent config, publish step) on top of a signal source that has a known, unfixed reliability gap.

Meanwhile, current production (`web/src/hooks/useChat.ts`, shipped `111c8fd`) already does everything the orchestrator does: Generic answers fully and deterministically (verified T1 tonight), a `SPECIALIST:`-prefixed suggestion is the offer signal, the client reads it and gates the specialist call on real human consent (`runDeepDive`), never auto-chains. Two agents, no tool-call machinery, already live, already stable except for the one caching-race caveat.

**Net assessment: the orchestrator, as validated, is not a fix for the actual bug. It's a more complex architecture that reproduces current behavior 1:1 while inheriting the same open defect.** This is worth saying plainly rather than building past it.

Two tracks follow. Track A fixes the real bug with a small, surgical change and keeps the current 2-agent architecture. Track B builds the validated orchestrator anyway, for a different reason (extensibility / AC2's port), with the caching race called out as a still-open, separate problem it does not solve. **Pick one before Build starts** — that decision is yours, not mine to make silently.

## 1. Track A (recommended) — fix the caching race directly, no new agent

### What's actually broken
`config.suggestions` fires as a platform-managed async job. When Agent Studio's per-query response cache resolves before that job finishes, the cached response can be replayed to a later identical (or near-identical) query with no suggestion attached — the offer never surfaces, with no error, no retry, no signal to the client that anything is missing.

### The fix
Stop depending on Agent Studio's async suggestion job for the één signal the UX actually needs (`SPECIALIST:` offer yes/no). Instead, have the client make its OWN synchronous, awaited second call — using the exact same prompt content already written and proven in `suggestions_generic.md` — right after Generic's answer completes, inside `runTurn`, before the turn is marked settled. This is the identical pattern the E2E test itself already used and validated 10/10 tonight (`callDirectAgent` in the test script): a plain, awaited `completions` POST, not a fire-and-forget platform job.

Concretely: `runTurn` currently does one `callWithRetry` to Generic and reads `genericResult.suggestions` (async, race-prone). The fix adds one more awaited call — after Generic's real answer text is in hand — to a lightweight classification request using `suggestions_generic.md`'s content as a one-shot prompt (question + Generic's real answer + real hits as context), and derives `deepDiveOffered`/`deepDiveQuery` from THAT response instead of the racy `config.suggestions` frame. Client controls the timing; there is no cache to race because the client isn't relying on the platform's own internal completion-caching to also finish a second job before returning.

### What gets modified
- `web/src/hooks/useChat.ts` — `runTurn`: add the second awaited classification call after the Generic leg succeeds, before `deriveOfferState`. `deriveOfferState`'s CONTRACT stays identical (still derives `deepDiveOffered`/`followUp`/`deepDiveQuery` from a `suggestions: string[]`-shaped input) — only the SOURCE of that array changes, from `genericResult.suggestions` to the new synchronous call's result. `extractDeepDiveOffer` is unchanged (same `SPECIALIST:` parsing contract).
- `scripts/agents/agentConfig.mjs` — `buildSuggestionsConfig`/`config.suggestions` on the live agent can be turned OFF (`enabled: false`) once the client-side classification call replaces it, so Agent Studio stops doing redundant work for a signal the client no longer reads. Or leave it on as a harmless no-op if disabling risks an unknown side effect — verify with a live check either way (Task A2 below).
- `scripts/agents/suggestions_generic.md` — becomes the client-side classification prompt instead of Agent Studio's `config.suggestions.system_prompt`. Same content, different consumer. May need a small edit: it currently assumes Agent Studio automatically supplies "the tool outputs" as context — the client-side version needs to pass Generic's real hits explicitly in the request.
- `web/src/lib/agentStudio.ts` — no change to frame parsing (the new call uses the exact same `parseCompletionStream`/`callCompletions` already used for Generic and Technical). The `2:` suggestions-frame handling can stay (harmless if `config.suggestions` is later disabled — it'll just never fire) or be removed once A is confirmed working, per Task A5 below.

### What gets thrown away
- Nothing structural. `types.ts`, `DiscoveryCard.tsx`, `agents.ts` are all untouched — they operate on `deepDiveOffered`/`followUp`/`deepDiveQuery`, which keep the exact same contract.
- Agent Studio's `config.suggestions` feature itself becomes unused by ACS (still a real platform feature, just not the one carrying this signal anymore) — this mirrors the ADR already on file (`wiki/decisions/2026-07-09-drop-tool-call-handoff-use-native-suggestions.md`) doing the same kind of walk-back, in the opposite direction.

### What gets rebuilt
- One new pure function (name TBD, e.g. `classifyOffer(query, answer, hits): Promise<string[]>`) that builds the classification request and calls `callCompletions`, returning a `suggestions`-shaped array so it slots into the existing `deriveOfferState` contract unchanged.

### Tasks
- **A1 (RED test first):** write a `useChat.test.ts` case: given a Generic answer and a real captured implementation-question hits fixture, `classifyOffer` returns an array containing a `SPECIALIST:`-prefixed string; given a design-question fixture, it does not. Use REAL captured content (a real answer text + real hits from tonight's E2E run or a fresh live probe), not synthesized text — same discipline as every other test in this repo.
- **A2:** live probe — confirm turning `config.suggestions.enabled` off on a `-dev` copy doesn't change any OTHER observed behavior (e.g., does the platform still emit a `2:` frame at all, does disabling it 500 anything). Verify via a raw curl/probe script before touching the live agent config.
- **A3:** implement `classifyOffer` + wire into `runTurn`. Keep `deriveOfferState` byte-for-byte unchanged.
- **A4:** re-run the SAME 10-question set from tonight's E2E test (5 design + 5 implementation), but this time specifically INCLUDING at least 2 repeated/near-duplicate query wordings on purpose (the exact condition that triggers the caching race) — confirm the offer still appears deterministically even on a repeated query. This is the actual regression test for the bug being fixed; tonight's test used deliberately unique wordings and so never exercised the race condition at all.
- **A5:** once A4 passes clean, decide whether to strip `config.suggestions` off the live agent config entirely (cleaner) or leave it dark or (leave frame parsing in `agentStudio.ts`, just unused). Recommend stripping it — dead config that could silently start mattering again if someone re-enables it later is exactly the kind of stale-mechanism risk this project has hit twice already (HANDOFF section, `[[FOLLOWUP:]]` sentinel).
- **A6:** live browser verification (real click-through, network tab) — the ONE test case never run for either track. Confirm the offer now survives a repeated question in a real browser session, not just the API-level probe.
- **A7:** deploy, verify live in the served bundle (grep method, same as every prior ACS deploy).

### Bug-fix / retest loop for this track
1. Write the RED test (A1) against real fixtures — must fail against current code (no `classifyOffer` exists yet).
2. Implement the minimum to pass.
3. Run full `web/` vitest suite (`npm test`) — must stay green, no regression in `extractDeepDiveOffer`/`deriveOfferState`/`buildTechnicalHistory`/history summarization tests.
4. Run A4's repeated-query live probe — this is the actual acceptance bar, not a green test suite. If the offer still doesn't survive a repeat, the fix is wrong; go back to step 2, don't ship a "tests pass but the real bug still happens" state (this project's own Cardinal Rule).
5. Only after A4 passes: A5 (retire the dead config) → A6 (browser) → A7 (deploy).

## 2. Track B — build the validated pure-orchestrator (if wanted for reasons beyond this bug)

Valid reasons this might still be worth building: (a) Algolia-Central2's Phase 1 port wants a genuinely model-driven router across more than 2 personas where hardcoding the routing logic client-side doesn't scale; (b) a standing platform capability worth having regardless of this specific bug. **Not valid: this bug** — see §0.

### What's actually being built
A third agent (orchestrator) that never answers itself — 2 `client_side` tools (`call_generalist`, `call_specialist`) — plus client code that intercepts each tool-call frame and calls the REAL target agent directly, streaming its real answer to the UI (the orchestrator's tool RESULT is fed back to it purely to decide the next hop, never shown to the user). This is exactly `docs/spikes/2026-07-09-pure-orchestrator-VERDICT.md`'s build plan, now with the E2E-validated real-data confirmation behind it.

### What gets modified
- `scripts/agents/agentConfig.mjs` — add a third `PERSONAS` entry (orchestrator) with `extraTools: [CALL_GENERALIST_TOOL, CALL_SPECIALIST_TOOL]` (flat `client_side` shape, proven live: `{name, type:"client_side", description ≤200 chars, inputSchema}`). Add a publish-step call (proven: `status:"published"` in the create/PATCH body is silently ignored — needs a separate `POST /agents/{id}/publish`).
- `web/src/lib/agentStudio.ts` — add `9:` tool-call frame handling (already scaffolded — `toolInvocations` exists in `ParsedCompletion` but nothing currently reads it) and the resume-call body builder (proven shape: assistant message with `parts:[{type:"tool-invocation", toolInvocation:{state:"result", toolCallId, toolName, args, result}}]`).
- `web/src/hooks/useChat.ts` — `runTurn` becomes: call orchestrator → intercept `call_generalist` tool call → call REAL Generic directly with the tool call's `query` arg → stream Generic's real answer to the UI segment → resume orchestrator with `{answer, offerSpecialist}` (where `offerSpecialist` is STILL sourced from Generic's real suggestion — §0's finding means this hop doesn't remove the caching race, it just relocates where the decision is consumed) → if orchestrator chains to `call_specialist`, that's the accept signal — either auto-run Technical (removes the human deep-dive gate this project has deliberately kept since 2026-07-02) or use the chain as confirmation that an offer SHOULD be shown, then still wait for human click (keeps the gate, but now the orchestrator hop bought nothing behaviorally). **This ambiguity needs Arijit's decision before Build (open question below) — the two options are materially different UX, and neither was tested tonight.**
- `web/src/types.ts` — `ChatTurn` needs either nothing (if the orchestrator hop is purely internal plumbing feeding the same `deepDiveOffered`/`deepDiveQuery` fields) or a new field if the human-gate is removed.

### What gets thrown away
- Nothing from current production code is deleted outright — Track B is additive (new agent, new client interception logic) layered in front of the existing Generic/Technical direct-call code, which stays as the thing the orchestrator's tool calls resolve to.
- The `config.suggestions` mechanism becomes redundant in a different way than Track A: still the signal source, just now consumed by the orchestrator instead of directly by `useChat.ts`. Same caching-race exposure either way (§0).

### What gets rebuilt
- `useChat.ts`'s `runTurn` — substantially, per above (a 3-hop client interception loop replaces the current 1-call-then-optional-gated-second-call flow).
- A disposable-to-permanent promotion of `scripts/spikes/agent-tool-handoff/e2e-orchestrator-validation.mjs`'s create/resume logic into real production code paths (`agentStudio.ts`/`useChat.ts`), not copy-pasted — the wire shapes are proven, the code structure isn't reusable as-is (it's a standalone test script, not app code).

### Tasks
- **B0:** Resolve the auto-chain-vs-human-gate open question (above) with Arijit before writing code — this determines whether B3/B4 below removes or preserves the deep-dive consent UX.
- **B1:** Add orchestrator persona + tools to `agentConfig.mjs`, publish step to `build_acs_agents.mjs`. RED test: tool schema shape assertions (mirrors Track A's discipline — assert on the exact flat shape, not the docs' rejected nested shape).
- **B2:** `agentStudio.ts` — wire real tool-call frame handling + resume body builder. RED test: feed a captured REAL tool-call frame (from tonight's `docs/spikes/2026-07-10-e2e-orchestrator-results.json` — real captured data, not synthesized) and assert `ParsedCompletion.toolInvocations` populates correctly.
- **B3:** `useChat.ts` — rewrite `runTurn` per B0's resolved UX. RED tests: the 3-hop flow calls Generic and Technical with the right args at the right times, per B0's decision.
- **B4:** Live spot-check ≥5 real turns through a `-dev` deploy, mixing design/impl questions AND at least 2 repeated-wording questions (same addition as Track A's A4 — confirms whether the caching race is ALSO present in this path, since it inherits the same signal source per §0).
- **B5:** Live browser verification (never run for this design at all yet).
- **B6:** Human-gated production flip (mirrors the original suggestions-migration's B9 — explicit go-ahead required, this touches live agent IDs the deployed app hardcodes).

### Bug-fix / retest loop for this track
Same shape as Track A: RED test against real captured data → implement → full local suite green → live spot-check (B4) is the real acceptance bar, not the test suite → only then browser (B5) → only then the human-gated live flip (B6). If B4 shows the caching race is STILL present (expected, per §0), that's not a regression to fix here — it's confirmation Track B needs Track A's fix layered on top of it regardless, or it ships with the same known defect production already has.
